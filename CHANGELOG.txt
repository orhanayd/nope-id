nope-id changelog
=================

1.1.0 - New ID formats, encoding helpers, and correctness fixes
+ feature: uuidv7() - time-ordered UUID v7 (RFC 9562), database-index friendly
+ feature: ulid() - spec-compliant 26-char ULID (Crockford Base32)
+ feature: monotonicFactory() - monotonic ULID generator with isolated per-instance state
+ feature: snowflakeFactory() / snowflake() / decodeSnowflake() - 64-bit distributed IDs returned as strings (BigInt-safe)
+ feature: objectId() / decodeObjectIdTime() - MongoDB ObjectId-compatible 24-char hex
+ feature: sqidsFactory() - reversible integer <-> short-string encoding (encode/decode), optional alphabet/minLength/blocklist
+ feature: defineId() - typed prefixed IDs (generate/is/parse) with TypeScript template-literal types
+ feature: isValidUUID() and isValidULID() - format validators
+ feature: two new test suites (idtypes, encoding); total tests raised from 244 to 307
+ feature: benchmarks for the new ID types added to npm run benchmark
+ change: new ID formats are secure-only - added to the ESM, CJS, and browser builds, intentionally not to nope-id/non-secure
+ change: all new generators are tree-shakeable named exports (import only what you use)
+ change: sortableId(size > 22) now pads with Crockford Base32 instead of urlAlphabet so the whole id stays in the documented alphabet
+ change: collisionProbability() now also returns the exact count via totalPossibleBigInt (added to the TypeScript types)
+ change: test:core/test:features/test:utils/test:non-secure scripts run with plain node and a main-module guard instead of node --test
+ change: bumped the nanoid devDependency (used only by the benchmark and randomness comparison, not shipped) to 5.1.11
+ change: more rigorous benchmark - best-of-5 trials plus a global warmup so neither library is penalized for being measured first; reproducible runs show nope-id winning all 5 core benchmarks vs nanoid 5.1.11
+ change: performance pass with no behavior change - uuid() ~+40%, shortId() ~+30%, monotonicFactory() ~+26%, ulid() ~+23%, customAlphabet() ~+18% (which also speeds up slugId()/shortId()), uuidv7() ~+14%, objectId() ~2x; achieved via precomputed hex/Crockford lookup tables, direct UUID formatting, ULID building with String.fromCharCode, and customAlphabet reading the byte pool directly. nopeid() also got a small local-index loop speedup (~+2-4%, which flipped the 10-char benchmark from a tie to a win); an earlier String.fromCharCode rewrite of nopeid() was measured and rejected as a regression.
+ fix: collisionProbability() returned wrong probabilityForBillion / safeCount / yearsFor1Percent for large spaces (the clamped value was used for the math); the statistical metrics now use the unclamped value
+ fix: random() called with a non-positive byte count no longer corrupts the internal pool offset (which could repeat already-issued bytes on the next call)
+ fix: the per-suite test scripts now exit non-zero on assertion failures instead of always exiting 0
+ fix: corrected the documented test count (CLAUDE.md said 200, README said 244) and the collisionProbability example output in the README

1.0.0 - Initial release
+ feature: nopeid() secure, URL-safe ID generator backed by a pooled CSPRNG (webcrypto.getRandomValues)
+ feature: customAlphabet() / customRandom() / random() with rejection sampling (no modulo bias)
+ feature: prefixedId(), sortableId() (monotonic, ULID-like), uuid() (v4), slugId(), shortId()
+ feature: getFingerprint(), distributedId(), generateMany(), nopeidAsync()
+ feature: isValid() (constant-time), collisionProbability() (BigInt), decodeTime()
+ feature: 14 pre-built alphabets; ESM + CommonJS + browser builds; full TypeScript definitions
