nope-id changelog
=================

1.4.0 - orderedId() 21-char Base58 sortable + monotonic, secureToken/apiKey/defineToken bearer-secret family, five correctness/security fixes
+ feature: orderedId() - a fixed-format, strictly-monotonic, lexicographically-sortable 21-char Base58 ID. Layout: 8-char timestamp + 5-char counter + 8-char random tail. Same shape as sparkid, with one extra char of unguessable randomness (~47 bits vs sparkid's ~41), clock-rewind clamp (Date.now() going backwards is clamped to the largest timestamp already emitted), synthetic time-bump on counter overflow (lastTime += 1 instead of a busy-wait loop), and a dedicated random char pool (rejection-sampled into Base58 codes at refill, ~8K calls per refill). Throughput on Node 22 is tied with sparkid within +-1.5% across 15 best-of-7 trials (~16M ops/sec on darwin/arm64). Ships in all three secure builds (ESM, CJS, browser) plus index.d.ts
+ feature: orderedId.parse(id) - decomposes a 21-char orderedId into { timestamp: Date, counter: number, random: string } with full Base58 validation (throws on length mismatch, non-Base58 chars including 0/O/I/l which the alphabet skips on purpose)
+ feature: orderedId.asciiBytes() - 21-byte ASCII representation (Base58 char codes in latin1, NOT packed binary). For a compact packed binary form see the v2.1 toBytes/fromBytes plan
+ feature: orderedId.many(count) - generate an array of `count` strictly-monotonic, lexicographically-sortable orderedId()s in one call. Reads the wall clock once at the start of the batch and again only every 4096 IDs ((i & 4095) === 4095), amortizing the per-ID Date.now() cost (~37 ns, about 63% of the per-call hot path on darwin/arm64) across the batch with no background timer. Roughly 1.7-1.8x the throughput of calling orderedId() in a loop (~28M vs ~16M ids/sec, fresh-process Welch on Node 22) and ~1.6x sparkid's per-call generateId(). orderedId() itself is a thin wrapper over a shared internal nextOrderedIdWithMs(ms), and many() reuses that same module-level cache state, so a batch leaves the generator strictly monotonic for any orderedId() that follows. Ordering is always exact (the counter separates same-millisecond IDs); an embedded timestamp may lag real time by however long it takes to emit up to ~4096 IDs (sub-millisecond in practice). count <= 0 returns []; count > 1_000_000 throws (ORDERED_MANY_MAX, same rationale as generateMany)
+ feature: secureToken(size = 48) - URL-safe bearer-secret generator. Unlike nopeid() (which serves substrings of a long-lived cached pool string for throughput), secureToken allocates a fresh local Buffer per call, fills it from webcrypto.getRandomValues(), maps each byte to the 64-char URL-safe alphabet via the bias-free (byte & 63) trick, then zeros the raw byte buffer before returning - so a memory dump cannot leak tokens that have not yet been requested. Minimum size 32 (throws below), default 48. The returned JavaScript string itself stays in V8 heap and cannot be zeroized (documented in JSDoc)
+ feature: apiKey(prefix = 'nope_live', size = 40) - thin prefixed wrapper over secureToken, e.g. apiKey('sk_live', 40) returns "sk_live_<40 chars>". Prefix must be a non-empty string without whitespace; size flows through to the secureToken minimum (>= 32). JSDoc recommends storing hashed (SHA-256) tokens, never the raw value
+ feature: defineToken(prefix, opts) - Stripe-style typed bearer-token helper returning { generate, is, parse }. Same generate/is/parse triple as defineId() but the body comes from secureToken (no future-token cache) and the alphabet is fixed to URL-safe 64-char for stability. Size minimum 32 (default 40); the is() check enforces exact body length so SessionToken.is('sess_short') returns false
+ feature: benchmarks/orderedid-vs-sparkid.bench.js - 15-trial median A/B harness specifically for orderedId vs sparkid throughput. Exposed via the new "benchmark:vs-sparkid" npm script so the "comparable throughput" claim in the README is reproducible on any machine
+ feature: README "Choose the right ID" decision table - 10-row guide mapping use cases (log/request ID, public URL ID, DB primary key, invite code, API key, session token, password-reset token, email verification, payment token) to the right primitive with size and reasoning. Plus a "Security profiles" section that calls out pooled-vs-unpooled cache characteristics for each family
+ feature: tests/ordered-id.test.js - 16 tests covering format (21-char Base58 alphabet), strict monotonicity under load and across a simulated clock rewind, parse round-trip preserving the random tail, parse rejection of wrong length / non-Base58 / non-string input, counter advancement within the same millisecond, timestamp boundary advancement, asciiBytes round-tripping through parse, and an orderedId.many() block (count/alphabet/distinctness, strict monotonicity across the 4096-ID clock-refresh boundary, shared-state interleaving with orderedId(), parsed-timestamp window, count edge cases)
+ feature: tests/secure-token.test.js - 23 tests covering secureToken default + custom sizes up to 1024, URL-safe alphabet, SECURE_TOKEN_MIN >= 32 floor, non-integer/NaN rejection, large-request cold path (size > MAX_POOL_SIZE), high-volume uniqueness, isolation from nopeid()'s pool, plus apiKey and defineToken (prefix non-empty + no-whitespace validation, default prefix nope_live, generate/is/parse triple, wrong-length-body rejection, custom separator)
+ change: random() now returns a fresh Buffer (Node) / Uint8Array (browser) copy via Buffer.from(pool.subarray(...)) instead of a shared view into the internal pool. The previous Buffer.subarray() return was a mutable view into a refillable buffer - callers who held onto the result and then triggered a pool refill would see their bytes change underneath them. Internal hot-path consumers (sortableId, ulid, monotonicFactory, uuidv7, objectId) use a new private randomView(n) that keeps the old zero-alloc view shape; only the public random() pays the copy
+ change: customAlphabet() and customRandom() validate at factory time: throws on empty alphabet, alphabets > 256 chars, duplicate characters (e.g. customAlphabet('aab', 21) was previously accepted but produced biased output), and non-Latin-1 characters (charCodeAt > 255 - the underlying Uint8Array silently truncated emoji and other multi-byte chars). Validation runs once per factory call, not per generated ID - the hot path is unchanged
+ change: customAlphabet(size > CPOOL_TARGET) and customRandom(size > CPOOL_TARGET) now allocate a one-shot local Buffer for the cold path instead of trying to satisfy the request from the cached 32 KiB pool. Previously a customAlphabet(...)(100000) call silently returned only ~32K chars because substring() did not bounds-check; the cold path now produces exact size chars with rejection sampling sized to the remaining count
+ change: customRandom() hot-path refill validates that the user-supplied getRandom() returns at least the requested byte count; throws "getRandom must return at least the requested number of bytes" instead of silently producing short or empty IDs. The check runs only at pool refill (once per ~32K char-codes generated), so per-call overhead is zero
+ change: collisionProbability(length, alphabetSize) now uses -Math.expm1((-n * (n - 1)) / (2 * possibleIdsExact)) instead of 1 - Math.exp(...). For very small probabilities (e.g. 21 chars on a 64-char alphabet) the old form rounded to literal 0 in IEEE-754 double precision; the new form returns the actual ~5.88e-21. The README example block now shows the real number instead of "0, rounds to 0"
+ change: defineId().is() now enforces exact body length: defineId('usr', { size: 21 }).is('usr_a') correctly returns false (previously returned true because the check only validated prefix + alphabet membership). defineId() also throws at factory time if size is not a positive integer
+ change: sortableId() clamps Date.now() to lastTime on clock rewind. Previously when the system clock went backwards (NTP correction, container resume, VM live-migration), the timestamp branch reset lastTime to the smaller value, breaking monotonicity and risking collision with IDs already returned. The new branch reuses the larger cached timestamp and continues advancing the counter on it. monotonicFactory() already had this clamp; sortableId() now matches. Non-secure sortableId() gets the same fix
+ change: sortableId(size) is marked as "Legacy API" in both the README heading callout and the JSDoc @deprecated tag (TypeScript users see strikethrough in IDE; runtime behavior is unchanged). New code should prefer orderedId() - strictly monotonic, no size knob, no truncation foot-guns. The "Sortable head-to-head: sparkid vs nope-id sortableId()" benchmark section is kept as-is because it remains a real natural-defaults comparison; the orderedId-vs-sparkid head-to-head lives in benchmarks/orderedid-vs-sparkid.bench.js and is summarized as "comparable throughput" in the README prose
+ change: distributedId(size) now throws "distributedId size must be an integer >= 16" when called with size < 16. Previously sizes 4-15 silently returned only the 4-char fingerprint plus separator, stripping the random tail entirely - which defeats the entire purpose of the function. README dropped the "distributed-safe / perfect for multi-node" phrasing in favor of "origin-tagged / process fingerprint" with the new minimum size called out
+ change: decodeObjectIdTime(id) validates against /^[0-9a-fA-F]{24}$/ instead of trusting parseInt to silently produce a partial number. decodeObjectIdTime('z'.repeat(24)) now throws "Invalid ObjectId" instead of returning new Date(NaN)
+ change: generateMany(count, size) caps count at 1_000_000 and throws "Count exceeds maximum batch size" above. Protects against accidental denial-of-service via untrusted input (e.g. body.count = 1e9). Below the cap behavior is unchanged
+ change: isValid() JSDoc clarified - it intentionally does not early-return on the first bad character (so it avoids the obvious first-bad-char timing leak), but it is not strictly constant-time in the cryptographic sense (Set.has() implementation depends on the JS engine). The README headline phrasing changes from "Timing attack prevention" to "Reduced timing-leak validators" to match. Runtime behavior is unchanged
+ change: README "API Token Generation" example in Real-World Examples switched from prefixedId('sk', 32) + nopeid(64) to apiKey('sk_live', 40) + secureToken(48) + defineToken('sess', { size: 48 }). The previous example used the pooled nopeid() string cache, which is exactly what the Security profiles section now correctly warns against for bearer secrets. README also propagates the change to the CommonJS variant of the same example
+ change: README "Database Primary Keys", "Time-series" and "Express.js Application" examples switched from sortableId() to orderedId() to match the new flagship recommendation. The "Choose the right ID" table also recommends orderedId() for sortable DB primary keys
+ change: README accuracy pass propagated to README.tr.md and README.ru.md in parallel - all three locales got the collisionProbability example fix, the API Token Generation example rewrite, the sortableId() Legacy callout, the Real-World Examples switch to orderedId, the distributedId softening, and the "Timing attack prevention" -> "Reduced timing-leak validators" change. Three files in lockstep, no locale drift

Otherwise the wording is "comparable performance" / "comparable throughput". orderedId() vs sparkid currently sits in the "comparable" bucket

+ change: nopeid() ~2.9x faster at the default 21-char size (~18M to ~52M ops/sec measured on Apple M2 / Node 21) and 1.4-7x across the rest of the curve, with the biggest wins at small sizes (size=4: ~21M to ~152M, ~7.2x; size=21: ~16M to ~47M, ~2.9x; size=64: ~10.5M to ~19M, ~1.8x; size=128: ~7.9M to ~11.4M, ~1.4x). Achieved by paying Buffer.toString('latin1')'s ~50 ns fixed cost ONCE per pool refill - the pre-translated 64 KiB pool is decoded to a flat one-byte string (cached as idPoolStr) and each call returns a single idPoolStr.substring(start, end). substring() on a flat one-byte parent is a V8 SlicedString (O(1), zero-copy) for sizes >= 13 and a tiny inline copy below that, both well under the fixed cost that used to dominate. Hot path is still CSPRNG-backed end to end; output remains byte-equivalent to the previous Buffer.toString path (entropy unchanged at log2(64) = 6 bits per char, 126 bits at default size 21)
+ change: same cached-string trick now powers customAlphabet() and customRandom() - each closure keeps its own pre-decoded char-code pool (one buf.toString per refill, substring per call). customAlphabet at the default hex/16 settings is ~4.7x faster; slugId() and shortId() benefit proportionally since they delegate to it
+ change: uuid() rewritten to a 4096-entry pre-formatted v4 pool - at refill time it draws 4096*16 fresh CSPRNG bytes, applies the v4 version + RFC 4122 variant bit patches per slot, writes 32 hex chars into a 144 KiB Buffer with hyphens pre-baked at slot positions 8/13/18/23, then one toString gives a flat parent string. Each call is just substring(start, start+36). Throughput goes ~7.4M to ~24.5M ops/sec, which is ~96% of native crypto.randomUUID (~26M, C++ binding) - the closest a JS implementation gets to native here. Per-call entropy is still 122 random bits (16 bytes minus the 6 fixed v4/variant bits)
+ change: monotonicFactory() ~4.6x faster (~3.3M to ~15.3M) by reusing a closure-scoped 26-byte Buffer scratch and, on the hot same-ms increment path, only rewriting the byte(s) that actually changed in the random suffix instead of re-encoding the full 26 chars. ulid() ~1.1x faster from the same scratch-reuse pattern (no per-call Uint16Array allocation)
+ change: sortableId() ~1.9x faster (~4.9M to ~9.2M) by writing directly into a module-level 22-byte SORT_BUF scratch (and a one-shot tail buffer for sizes > 22) instead of building the timestamp via prepend-style ConsString churn + a separate randomPart concatenation
+ change: prefixedId() ~2.6x faster (~14M to ~37M) and generateMany(100) ~3.0x faster (~172K to ~512K) - both compose on nopeid() and inherit the speedup transparently; no source changes needed in those functions
+ change: bytesToUuid() rewritten to write 32 hex char codes into the module-level UUID_BUF then one toString('latin1'), instead of 16 byteToHex string lookups joined by 4 ConsString allocations. Used internally by uuidv7() and is the formatting backbone for the new uuid() pool
+ change: pre-translated CROCKFORD_CODES table is now Uint8Array (all 32 Crockford chars fit in a byte) instead of Uint16Array - lets ulid()/sortableId()/monotonicFactory() write directly into Buffer/Uint8Array scratches and finish with toString('latin1') / TextDecoder.decode(), the same shape as nopeid()'s hot path

1.2.0 - 2x-5x faster nopeid(), browser/non-secure parity, auto-updating README benchmarks
+ change: nopeid() ~2.8x faster at the default 21-char size (~6.7M to ~18.9M ops/sec) and up to ~4.9x at 64 chars (~2.7M to ~13.3M). Achieved via three combined changes: a dedicated id-pool whose CSPRNG bytes are translated to alphabet character codes in place on refill; a precomputed 64 KiB Uint16Array table mapping every 16-bit value to its two translated alphabet codes so the refill loop runs at half the iteration count; and a single Buffer.toString('latin1') per call instead of ~21 ConsString allocations from per-character concatenation. Hot path is still CSPRNG-backed end to end, output still carries 126 bits of entropy
+ change: nope-id is now the fastest CSPRNG-backed JS library for URL-safe full-alphabet IDs at equal entropy. Beats uid/secure (~9.4M ops/sec, smaller alphabet/lower entropy) for the first time. Native crypto.randomUUID() remains the fastest for plain v4 UUIDs and we continue to recommend it for that specific case
+ change: prefixedId() benefits proportionally (~7.2M to ~16M ops/sec) because it composes on top of nopeid()
+ change: index.browser.js gets the same strategy adapted to the browser - Uint8Array + cached TextDecoder('latin1').decode() in place of Buffer.toString('latin1'). The 16-bit batch refill table is endian-agnostic by construction (each output byte still derives from its own input byte)
+ change: non-secure variants (nope-id/non-secure ESM + CJS) now build IDs in a Uint8Array of pre-translated codes and call String.fromCharCode.apply() once instead of per-character concatenation - ~3.4M to ~4.2M ops/sec at the default size
+ change: isValid() with the default urlAlphabet is ~16x faster (~442K to ~7M ops/sec) by hoisting a single Set at module level instead of allocating a fresh 64-element Set on every call. Custom alphabets keep the per-call Set
+ change: slugId(size) and shortId(size) at non-default sizes are ~1.4x faster - they now delegate to the cached factory (whose closure already accepts a size argument), instead of allocating a brand-new customAlphabet factory on each call
+ change: replaced Array.from(bytes, b => b & 31) with explicit for-loops in sortableId() and monotonicFactory() so the inner arrow does not depend on V8 hoisting. No measurable steady-state delta, just one fewer engine assumption
+ change: removed unused urlAlphabetLookup constant from index.js, index.cjs and index.browser.js (became dead code after the in-place pool translation landed). Resolves a Biome noUnusedVariables lint
+ feature: benchmark --json mode (node benchmarks/benchmark.js --json) emits structured results to stdout for tooling - meta (date, node, platform, arch, runner) plus grouped numbers (comparison, uuid, ulid, vs_others, extras). The human-readable colored output is suppressed in --json mode; the default invocation is unchanged

Six regions are bracketed by invisible HTML comment markers (bench:headline, bench:comparison-table, bench:uuid-table, bench:ulid-table, bench:speed-vs-entropy-table, bench:extras-table) plus a bench:meta line; the script throws on missing keys so a parser regression fails loudly instead of writing undefined into the markdown
+ feature: GitHub Actions workflow .github/workflows/benchmark-on-pr.yml runs on every pull_request against main, runs the full test gate (npm test, test:randomness, test:types, test:browser), then runs the JSON benchmark and commits the refreshed README back to the PR branch. actions/setup-node pinned to a SHA (v6.4.0) on node-version 26; concurrency cancels stale runs; fork PRs are skipped (no token to push back)
+ feature: tests/types.test.ts exercises every public export with explicit type annotations, compiled by tsc --noEmit. Catches index.d.ts drift at CI time
+ feature: tests/browser.test.js imports index.browser.js and validates 17 behaviors via the existing test-utils framework (default size, edge cases, cold path, uniqueness, prefixedId, sortableId+decodeTime round-trip, uuid v4 format, alphabet identity). Runs under Node 18+ using the global crypto and TextDecoder
+ feature: new npm scripts - benchmark:json, benchmark:update-readme, test:types, test:browser, test:all (composes the four CI gates locally)
